# SmbPhotoLibrary 更新日志

## v1.0.0 正式版（2026-05-16）

从测试版到正式版，重点修复了视频播放相关的多个严重问题。

---

## 🐛 问题修复

### P0 · 视频播放完全失效（ProtocolException + 416 错误）

#### 问题 1：HTTP Chunked Encoding 格式错误

**错误日志：**
```
java.net.ProtocolException: Expected leading [0-9a-fA-F] character but was 0x19
```

**根因：** 当文件大小未知时，代理设置 `Transfer-Encoding: chunked`，但 `streamData()` 直接发送原始二进制数据，没有按 HTTP chunked 格式封装。ExoPlayer 把视频数据当 chunk header 解析。

**修复：**
- 新增 `useChunkedEncoding` 参数区分模式
- 新增 `writeChunk()` 函数，正确输出 `[size_hex]\r\n[data]\r\n`
- 结束时发送 `0\r\n\r\n` 终止块
- 只在全部数据写入后统一 flush

**涉及文件：** `SmbProxyServer.kt`

---

#### 问题 2：SMB 文件大小读取失败（size=0）

**错误日志：**
```
HttpDataSource$InvalidResponseCodeException: Response code: 416
```

**根因：** `SmbManager.openInputStreamWithSize()` 直接返回传入的 `knownSize` 参数，没有从 SMB 文件对象获取实际大小。当 `knownSize=0` 时，所有视频流都变成 0 字节。

**修复：**
```kotlin
val fileSize = try {
    val fileInfo = smbFile.fileInformation
    fileInfo.standardInformation.endOfFile
} catch (e: Exception) {
    knownSize  // 后备方案
}
```

**涉及文件：** `SmbManager.kt`

---

#### 问题 3：Per-chunk Flush 导致数据丢失

**根因：** 每次小块写入后立即 `out.flush()`，当 socket buffer 满报 Broken pipe 时，已 flush 数据已推送但后续数据丢失。客户端收到残缺 HTTP 响应。

**修复：** 移除所有 per-chunk flush，只在整次请求结束时统一 flush。

**涉及文件：** `SmbProxyServer.kt`

---

### P0 · 应用假死卡顿（线程阻塞 30 秒）

#### 问题：tryReconnect() 持有锁时间过长

**错误日志：**
```
Long monitor contention 31.046s on java.lang.Object@xxxxx
```

**根因：** `SmbConnectionPool.tryReconnect()` 在持有锁的状态下执行 `establishConnection()` 网络 I/O（耗时 30 秒），阻塞所有其他线程。

**修复：**
```kotlin
fun tryReconnect(): Boolean {
    val config = fullConfigWithPassword ?: currentConfig ?: return false
    if (isConnected) return true              // 无锁快速路径
    val gotLock = lock.tryLock(5, TimeUnit.SECONDS)
    if (!gotLock) return false                // 5 秒超时，快速失败
    closeInternal()                           // 锁内（很快）
    lock.unlock()
    establishConnection(config)               // 【锁外】网络 I/O，不阻塞
}
```

**优化点：**
- `lock` 从 `Any()` → `ReentrantLock()`
- `acquireExistingConnection()` 移除 synchronized，只做 volatile 只读判断
- 所有方法改用 `lock.withLock {}`

**涉及文件：** `SmbConnectionPool.kt`

---

### P2 · 系统导航兼容性

#### 问题：Android 14 手势导航冲突

**修复：** 在 `AndroidManifest.xml` 添加 `android:enableOnBackInvokedCallback="true"`

**涉及文件：** `AndroidManifest.xml`

## 🔧 技术细节

### HTTP Chunked Transfer Encoding 正确格式

```
HTTP/1.1 200 OK
Transfer-Encoding: chunked
Content-Type: video/mp4

[size_in_hex]\r\n
[data]\r\n
...（重复）...
0\r\n
\r\n
```

### SmbConnectionPool 锁优化对比

| 指标 | 修复前 | 修复后 |
|-----|-------|-------|
| tryReconnect() 持锁时间 | ~30s（网络 I/O） | <1s（仅资源清理） |
| acquireExistingConnection() | synchronized 阻塞 | volatile 只读 O(1) |
| 锁类型 | `Any()` | `ReentrantLock()` |
| 其他线程等待 | 最多 30 秒阻塞 | 最多 5 秒超时 |
